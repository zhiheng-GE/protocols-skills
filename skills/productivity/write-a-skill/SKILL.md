# Writing Skills

## 流程

1. **收集需求** — 询问用户：
   - 这个 skill 覆盖什么任务或领域？
   - 具体使用场景有哪些？
   - 是否需要可执行脚本，还是只需要指令？
   - 是否有参考资料要包含？

2. **起草 skill** — 创建：
   - 简洁的 `SKILL.md`；
   - 如果内容超过约 500 行，拆出额外参考文件；
   - 如果有确定性操作，加入工具脚本。

3. **与用户评审** — 展示草稿并询问：
   - 是否覆盖使用场景？
   - 是否遗漏或不清楚？
   - 哪些部分应更详细或更简短？

4. **安装/链接** — 根据目标 agent 说明如何安装到 `~/.claude/skills` 或 `~/.codex/skills`。

## Skill 结构

最小结构：

```text
my-skill/
└── SKILL.md
```

复杂结构：

```text
my-skill/
├── SKILL.md
├── references/
│   └── topic.md
└── scripts/
    └── helper.sh
```

## `SKILL.md` 模板

```markdown
---
name: my-skill
description: 何时使用这个 skill。包含用户可能说出的触发短语。
---

# My Skill

简短说明目标。

## 流程

1. ...
2. ...

## 规则

- ...

## 输出

...
```

## 写作原则

- **触发清楚**：description 说明何时使用，而不只是说它是什么。
- **指令可执行**：告诉 agent 做什么、按什么顺序、何时停止。
- **渐进披露**：主文件只放核心流程；长参考内容拆到单独文件。
- **少即是多**：不要把通用常识塞进 skill。
- **保留技术字面量**：命令、路径、API、错误消息不要翻译或改写。
- **可验证**：说明成功标准或输出格式。

## 常见错误

- description 太泛，导致误触发；
- 主文件过长，模型难以抓住流程；
- 把脚本能稳定完成的事写成自然语言步骤；
- 没有说明失败时如何处理；
- 缺少示例输出。
