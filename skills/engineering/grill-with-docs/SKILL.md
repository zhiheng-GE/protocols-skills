## 领域感知

探索代码库时，同时寻找现有文档。

### 文件结构

多数仓库只有一个上下文：

- `CONTEXT.md`：项目领域语言、术语、关系和歧义。
- `docs/adr/`：架构决策记录。

如果仓库已有等价文件（例如 `docs/context.md`、`docs/decisions/`），沿用现有结构，不要强行迁移。

### 如何使用 `CONTEXT.md`

- 先读它，理解项目用语。
- 追问时使用其中的术语。
- 如果用户引入新概念，判断它是否应加入语言表。
- 如果发现一个词有歧义，追问直到能写下定义和避免用语。
- 不要为了显得正式而发明术语；只记录能让未来 agent 更简洁、更准确的词。

### 如何使用 ADR

当计划涉及不可逆或高影响决策时，检查是否已有 ADR：

- 技术选型；
- 模块边界或接缝；
- 数据模型；
- 部署/运行约束；
- 安全、隐私、成本或性能取舍。

若当前讨论形成了新决策，准备新增 ADR。若改变已有决策，准备更新或 supersede 原 ADR。

## 追问方式

目标不是问很多问题，而是消除会导致返工的未知数。

优先追问：

1. **目标和非目标**：我们到底要改变什么？明确不做什么？
2. **用户和场景**：谁会用？成功路径是什么？失败路径是什么？
3. **领域语言**：这个概念在项目里叫什么？是否已有同义词或歧义？
4. **现有系统**：哪些模块、接口、数据流会被触碰？
5. **约束**：兼容性、迁移、性能、安全、部署、成本。
6. **验收**：怎么知道完成？哪些测试或检查必须通过？
7. **文档影响**：是否需要更新 `CONTEXT.md` 或 ADR？

每次提问格式：

```text
问题：...
为什么重要：...
我的推荐：...
```

## 写文档

当答案稳定后，直接更新文档，而不是另开总结。

### 更新 `CONTEXT.md`

只加入稳定、有复用价值的领域语言：

- 术语；
- 定义；
- 避免使用的近义词；
- 关系；
- 已澄清的歧义。

### 新增 ADR

当形成架构决策时，在 `docs/adr/` 新增递增编号文件：

```text
docs/adr/0002-short-kebab-title.md
```

使用本 skill 目录中的 `ADR-FORMAT.md`。

## 结束条件

当以下条件都满足时结束：

- 目标、非目标和验收标准清楚；
- 关键设计分支已决策；
- 术语无明显歧义；
- 需要的 `CONTEXT.md` / ADR 更新已完成；
- 可以把计划交给另一个 agent 实施而无需再做产品或架构决策。
