# Triage

通过一套小型分流角色状态机，在项目问题追踪器中推进问题/工作项。

在 triage 期间发布到问题追踪器的每条评论或 issue 都必须以此免责声明开头：

```text
> *This was generated by AI during triage.*
```

## 参考文档

- [AGENT-BRIEF.md](AGENT-BRIEF.md) — 如何写持久的 agent brief。
- [OUT-OF-SCOPE.md](OUT-OF-SCOPE.md) — `.out-of-scope/` 知识库如何工作。

问题追踪器和分流标签词汇应已提供；如果没有，请运行 `/setup-matt-pocock-skills`。

## 角色

两个 **分类** 角色：

- `needs-triage` — 新进入或尚未判断的问题。
- `out-of-scope` — 明确不做的问题，原因应写入评论或 `.out-of-scope/`。

三个 **行动** 角色：

- `needs-info` — 缺少关键信息，需要用户或报告者补充。
- `ready-for-agent` — 可以由在线 agent 继续处理。
- `ready-for-afk` — 可交给离线/异步 agent 独立执行。

每个 issue 同一时间只应有一个分流角色。真实 label 字符串由 `docs/agents/triage-labels.md` 映射。

## 流程

### 1. 读取 issue

读取正文、评论、label、相关代码上下文和已有链接。不要只看标题。

### 2. 判断范围

如果明显不属于项目范围：

- 应用 `out-of-scope`；
- 解释原因；
- 如果这是可复用规则，更新 `.out-of-scope/` 知识库。

### 3. 判断信息是否足够

如果无法重现、验收或决策，应用 `needs-info`，并只问最少必要问题。

问题应具体，例如：

- “请提供复现步骤和实际错误输出”；
- “这个行为应发生在哪个用户角色下？”；
- “请确认预期结果 A/B 哪个正确”。

### 4. 判断是否 ready

`ready-for-agent`：

- 目标明确；
- 验收标准明确；
- 相关代码区域可定位；
- 不需要产品/架构决策。

`ready-for-afk` 要求更高：

- 可以无人值守执行；
- 有完整 agent brief；
- 测试/验证命令明确；
- 失败时有清晰回退或汇报方式。

### 5. 写 agent brief

如果 issue 可执行，写清：

- 背景；
- 要实现/修复的行为；
- 相关模块；
- 非目标；
- 验收标准；
- 测试命令；
- 风险和注意事项。

## 输出

汇报：

- issue 编号/链接；
- 原分流角色和新分流角色；
- 采取的操作；
- 仍需人工回答的问题。
