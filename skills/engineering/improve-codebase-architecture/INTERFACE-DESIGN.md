# 接口设计

好接口让调用方少知道。它不是更短的函数签名，而是更少的使用知识。

## 好接口

- 名称使用项目领域语言。
- 输入表示调用方真实拥有的信息。
- 输出表示调用方真正需要的结果。
- 错误模式明确、数量少。
- 默认值和顺序隐藏在模块内部。
- 可通过公共行为测试。

## 坏接口

- 要求调用方按隐含顺序调用多个函数。
- 泄漏内部状态或外部 API 细节。
- 返回半成品数据，要求调用方继续拼装。
- 需要调用方知道缓存、重试、分页、事务等实现策略。
- 每个调用方都要写同一段保护逻辑。

## 评审问题

- 调用方必须知道哪些东西？哪些可以隐藏？
- 如果实现完全替换，接口是否仍然成立？
- 这个接口能不能支持一个真实行为测试？
- 它位于正确接缝吗？
